# AvaTracker Mobile — контракт API бэкенда

> ⚠️ Разделы 1–8 ниже описывают **изначально предложенный** контракт.
> Фактически развёрнутый бэкенд отличается — клиент v1.0.0 интегрирован
> с ним по разделу **0 «Реальный бэкенд»** ниже. При расхождении — раздел 0 главный.

---

## 0. Реальный бэкенд (проверено на живом сервере)

Мобильное приложение использует **три базы** на одном хосте:

| Назначение | База | Авторизация |
|---|---|---|
| Авторизация + профиль | `https://avatracker.online/api/mobile` | **SimpleJWT** (`Authorization: Bearer <access JWT>`) |
| QR-скан и точки отметки | `https://avatracker.online/api/qr/…` | Bearer-токен (тот же JWT / тест-токен) |
| Данные (табель, аналитика, профиль в тест-режиме) | `https://avatracker.online/api/v1` | тот же токен |

### 0.1. Отметка по QR

`POST /api/qr/scan/` — тело:

```json
{
  "iin": "990101300123",
  "qr_id": "park_karaganda_main_entrance",
  "employee_latitude": 49.80471,
  "employee_longitude": 73.10952,
  "accuracy_meters": 12,
  "is_mock_location": false,
  "scanned_at": "2026-06-29T12:24:37+05:00",
  "photo": "<JPEG фронтальной камеры в base64>"
}
```

Обязательны все поля, кроме `accuracy_meters`. **`photo`** — снимок лица:
сервер сверяет его с базовым фото сотрудника прямо при скане (отдельного
face-verify нет). Экран FaceID в приложении только делает снимок и отдаёт его
в этот запрос. Скан незарегистрированного QR → `404 {"success": false,
"message": "QR-локация не найдена."}` (проверка QR идёт до обработки фото).

**Точка отметки** `GET /api/qr/{qr_id}/` → `{id, qr_id, park_id, park_name,
latitude, longitude, radius_meters, is_active, …}`; нет точки → 404. Клиент
делает этот запрос **до** FaceID: незарегистрированную/отключённую точку
отсекает сразу, не заставляя делать селфи. Список/создание точек —
`GET/POST /api/qr/` (админские, мобилке не нужны).

Перед реальным сканированием в backend должна существовать активная точка,
созданная через `POST /api/qr/` с `qr_id`, `park_id`, координатами и
`radius_meters`. Статичный QR содержит этот `qr_id`.

- Ошибки — DRF-формат: `{"detail": "..."}` или ошибки по полям
  `{"phone": ["Обязательное поле."]}`. Тексты показываются пользователю как есть.
- Токены: SimpleJWT — ответ `{"access": "...", "refresh": "..."}`.
  Refresh: `POST /api/mobile/auth/token/refresh/` тело `{"refresh": "..."}` → `{"access": "..."}`.
- `dial_code` хранится отдельным полем от `phone`, **но формат `phone`
  зависит от шага** (проверено рабочими запросами):
  - `login/`, `register/`, `register/resend/`, `password-reset/request/`:
    `phone` = 10 цифр (`7714322337`) + `dial_code = "+7"`.
  - `register/verify/`, `password-reset/verify/`: `phone` = **полный** номер
    с кодом страны без «+» (`77714322337`), поля `dial_code` нет.
  Клиент поддерживает Казахстан (`+7`) и Узбекистан (`+998`); backend должен
  принимать оба dial code.

### 0.2. Эндпоинты (база `…/api/mobile`)

| Метод и путь | Тело запроса | Ответ |
|---|---|---|
| `POST /auth/register/` | `{phone, dial_code, iin, password, password2}` | отправляет SMS (пароль задаётся здесь) |
| `POST /auth/register/verify/` | `{phone, code}` | подтверждает регистрацию → JWT |
| `POST /auth/register/resend/` | `{phone, dial_code}` | повторная отправка SMS |
| `POST /auth/login/` | `{phone, dial_code, password}` | `{access, refresh}` |
| `POST /auth/token/refresh/` | `{refresh}` | `{access}` |
| `POST /password-reset/request/` | `{phone, dial_code}` | отправляет SMS |
| `POST /password-reset/verify/` | `{phone, code}` | `{reset_token}` (код не сжигается) |
| `POST /password-reset/confirm/` | `{reset_token, password, password2}` | пароль изменён |
| `GET /profile/me/` | — (JWT) | профиль текущего сотрудника |

Поведение клиента:

1. **Регистрация**: `register/` (с паролем) → SMS → `register/verify/` (только код).
   Если verify вернул `access/refresh` — входим сразу; иначе клиент делает
   `login/` тем же паролем. После сессии профиль грузится из `GET /profile/me/`.
2. **Вход**: `login/` → сохраняем JWT → `GET /profile/me/` → главный экран.
3. **Сброс**: `request/` → `verify/` (получаем `reset_token`, показываем форму
   нового пароля) → `confirm/` (`reset_token` + новый пароль ×2) → экран входа.
4. **Профиль**: реальный режим — `GET /api/mobile/profile/me/`;
   тест-режим (`TEST_BEARER_TOKEN`, 40-символьный токен `/api/v1`) —
   `GET /api/v1/employees/{iin}/`.
5. Смена пароля: `POST /auth/change-password/` (Bearer JWT), тело
   `{current_password, new_password, new_password_confirm}`. FaceID-сверки
   отдельным запросом нет — фото лица уходит в `/qr/scan/`.
6. `GET /profile/me/` парсится толерантно: `position_name`/`division_name`
   предпочтительнее GUID; UUID в профиле не показывается.

## 1. Общие правила

### 1.1. Авторизация

JWT-пара: `access_token` (Bearer) + `refresh_token`.

- Все эндпоинты, кроме раздела «Публичные», требуют заголовок
  `Authorization: Bearer <access_token>`.
- При ответе **401** на защищённый эндпоинт клиент автоматически вызывает
  `POST /mobile/auth/refresh` и повторяет исходный запрос **один раз**.
  Если refresh не удался — принудительный выход на экран входа.
- Рекомендуемые сроки жизни: access ~30 мин, refresh ~30 дней (ротация при каждом refresh).

Публичные (без токена):

```
POST /mobile/auth/register/send-code
POST /mobile/auth/register/verify
POST /mobile/auth/login
POST /mobile/auth/password/forgot
POST /mobile/auth/password/verify-code
POST /mobile/auth/password/reset
POST /mobile/auth/refresh
```

### 1.2. Формат ошибок

Любая ошибка — HTTP 4xx/5xx с телом:

```json
{
  "success": false,
  "message": "Текст ошибки на русском — показывается пользователю как есть",
  "error_code": "MACHINE_CODE"        // опционально, для бизнес-отказов
}
```

Клиент берёт текст из `message` (фолбэк: `error`, затем `detail`) и показывает его
без изменений — **все message писать по-русски, человекочитаемо**.

⚠️ Если на 401 не передать `message`, клиент покажет «Сессия истекла. Войдите заново».
Поэтому для бизнес-ошибок входа/кодов использовать **400**, а 401 оставить только
для невалидного/протухшего токена.

### 1.3. Форматы данных

| Поле | Формат | Пример |
|---|---|---|
| `phone` | E.164 Казахстан, без пробелов | `"+77001234567"` |
| `iin` | строка, ровно 12 цифр | `"990101300123"` |
| `code` | SMS-код, 4 цифры | `"1234"` |
| `password` | ≥ 6 символов, без пробелов | `"secret123"` |
| дата-время | ISO-8601 с локальным смещением | `"2026-06-12T16:22:00+05:00"` |
| `month` | `YYYY-MM` | `"2026-06"` |
| минуты | целые числа (`*_minutes`) | `480` |

### 1.4. Индивидуальный график

- Время начала/окончания смены индивидуально для сотрудника и дня.
- `/mobile/attendance/timesheet` обязан возвращать `work_start` и `work_end` для
  каждого рабочего дня; при наличии обеда — `lunch_start` и `lunch_end`.
- `worked_minutes` рассчитывает сервер по фактическим отметкам с учетом обеда.
- Клиент не предполагает фиксированный график 09:00–18:00 и не вычисляет
  рабочие/выходные дни по шаблонам 2/2, 6/0 и т.п.

### 1.5. SMS-коды и пароли

- Код: 4 цифры, TTL ~5 минут, хранится хэшем.
- Не более **5 попыток** ввода кода, повторная отправка не чаще **1 раза в 60 с**.
- `verify-code` (шаг сброса) **не сжигает код** — следом придёт `reset` с тем же кодом.
- Код сжигается после успешного `register/verify` или `password/reset`.
- Пароль: минимум 6 символов без пробелов, хранить хэшем (argon2/bcrypt).
- Flutter-приложение пароль не сохраняет: пароль отправляется только в запросах
  login/register/reset/change. Локально хранятся только токены и данные сессии
  (см. §1.7).

### 1.6. SMSC.kz

SMSC.kz вызывается только из backend. Логин/пароль запрещено передавать в
Flutter-приложение или хранить в репозитории. В `.env` проекта
`avatracker-back` добавить:

```dotenv
SMSC_LOGIN=
SMSC_PASSWORD=
SMSC_SENDER=
SMSC_API_URL=https://smsc.kz/rest/send/
```

Backend отправляет POST на `SMSC_API_URL` с `login`, `psw`, `phones`, `mes`,
`charset=utf-8` и ожидает JSON-ответ. Вместо пары login/password SMSC.kz также
поддерживает API key. Официальная документация:
https://smsc.kz/api/http/send/

### 1.7. Локальное хранение в мобильном приложении

Клиент использует `flutter_secure_storage`: Android Keystore /
EncryptedSharedPreferences на Android и Keychain на iOS.

Локально сохраняются:

- `access_token`;
- `refresh_token`;
- `iin`;
- `phone`;
- `employee_json` — базовый профиль сотрудника для быстрого старта.

Локально **не** сохраняются:

- пароль;
- SMS-коды;
- FaceID-снимки;
- одноразовые FaceID-токены;
- `device_id`.

`device_id` и отдельный `face_verification_token` в API не передаются.
При отметке клиент отправляет Bearer-сессию, ИИН, `qr_id`, координаты,
точность GPS, признак mock-location, время и JPEG-снимок лица в base64.
Сервер проверяет сотрудника, лицо, QR-точку и расстояние в одном запросе
`POST /api/qr/scan/`.

## 3. Авторизация

При запуске приложение **не** запрашивает FaceID. Оно проверяет наличие
`access_token`/`refresh_token`, затем подтягивает `GET /profile/me/`.
FaceID выполняется только после сканирования QR и только перед отправкой
`POST /api/qr/scan/`.

### 3.0. Последовательность регистрации

1. Пользователь вводит телефон, ИИН, пароль и подтверждение пароля.
2. Клиент локально проверяет формат и вызывает
   `POST /mobile/auth/register/send-code`, передавая только `phone` и `iin`.
   Пароль на первом шаге на сервер не отправляется.
3. Backend проверяет сотрудника, активность, допустимость привязки телефона,
   отсутствие существующего аккаунта и rate limit; затем создает временный
   SMS challenge и отправляет код.
4. Клиент открывает экран SMS, принимает четыре цифры и автоматически вызывает
   `POST /mobile/auth/register/verify`.
5. Во втором запросе клиент отправляет `phone`, `iin`, `code` и `password`.
6. Backend атомарно проверяет challenge, хэширует пароль, создает аккаунт,
   сжигает SMS-код и возвращает JWT-сессию.
7. Клиент сохраняет только access/refresh-токены, ИИН, телефон и профиль в
   secure storage. Пароль и SMS-код локально не сохраняются. После успешного
   ответа роутер открывает экран QR-сканера.

### 3.1. `POST /mobile/auth/register/send-code`

Регистрация, шаг 1. Сервер проверяет, что сотрудник с таким ИИН существует
в базе AvaTracker, активен, и телефон не занят другим аккаунтом, затем шлёт SMS.

Важно: backend должен проверить, что переданный телефон разрешено привязать к
этому сотруднику. Безопасный базовый вариант — нормализованный `phone` должен
совпадать с телефоном сотрудника в AvaTracker. Нельзя разрешать регистрацию по
чужому ИИН на произвольный подтвержденный SMS-номер.

Запрос:

```json
{
  "phone": "+77001234567",
  "iin": "990101300123"
}
```

Ответ 200:

```json
{
  "success": true,
  "message": "SMS code sent"
}
```

Ошибки:

| HTTP | message (пример) | Когда |
|---|---|---|
| 404 | `Сотрудник с таким ИИН не найден` | ИИН отсутствует в базе |
| 400 | `Доступ запрещен. Сотрудник неактивен` | `active = false` |
| 409 | `Этот номер уже зарегистрирован. Войдите по паролю` | телефон занят |
| 429 | `Повторная отправка возможна через 60 секунд` | rate limit |

### 3.2. `POST /mobile/auth/register/verify`

Регистрация, шаг 2. Проверка кода, создание аккаунта с паролем, выдача сессии.

Запрос:

```json
{
  "phone": "+77001234567",
  "iin": "990101300123",
  "code": "1234",
  "password": "secret123"
}
```

Ответ 200 (единый «ответ-сессия», также используется login):

```json
{
  "success": true,
  "access_token": "eyJhbGciOi...",
  "refresh_token": "eyJhbGciOi...",
  "employee": { /* объект сотрудника, см. §4.1 */ }
}
```

Требования клиента: `success = true` **и оба токена обязательны**.
`employee` крайне желателен (иначе клиент сделает дополнительный
`GET /employees/{iin}/`).

Если `employee` не возвращается, нужно вернуть `iin` верхнего уровня. Без
`employee` и без `iin` клиент не сможет сохранить сессию.

Ошибки:

| HTTP | message |
|---|---|
| 400 | `Неверный SMS-код` |
| 400 | `Код истек. Запросите новый` |
| 400 | `Превышено количество попыток. Запросите новый код` |
| 400 | `Пароль: минимум 6 символов, без пробелов` |

### 3.3. `POST /mobile/auth/login`

Запрос:

```json
{
  "phone": "+77001234567",
  "password": "secret123"
}
```

Ответ 200 — тот же «ответ-сессия», что в §3.2.

⚠️ **`employee` (или хотя бы поле `"iin"` верхнего уровня) в ответе обязателен**:
при входе на новом устройстве клиенту больше неоткуда узнать ИИН — без него
вход завершится ошибкой «Сервер не вернул ИИН сотрудника».

Ошибки:

| HTTP | message |
|---|---|
| 404 | `Аккаунт не найден. Зарегистрируйтесь` |
| 400 | `Неверный номер телефона или пароль` |
| 400 | `Доступ запрещен. Сотрудник неактивен` |

### 3.4. `POST /mobile/auth/password/forgot`

Запрос:

```json
{ "phone": "+77001234567" }
```

Ответ 200: `{ "success": true, "message": "SMS code sent" }`

Ошибки: 404 `Аккаунт с этим номером не найден`, 429 — как в §3.1.

### 3.5. `POST /mobile/auth/password/verify-code`

Проверка кода сброса **без смены пароля** (клиент показывает форму нового пароля
только после успешной проверки). Код при этом не инвалидируется.

Запрос:

```json
{
  "phone": "+77001234567",
  "code": "1234"
}
```

Ответ 200: `{ "success": true }`

Ошибки: 400 `Неверный SMS-код` / `Код истек. Запросите новый` /
`Превышено количество попыток. Запросите новый код`.

### 3.6. `POST /mobile/auth/password/reset`

Финал сброса: тот же код + новый пароль. Код сжигается, все refresh-токены
аккаунта рекомендуется отозвать.

Запрос:

```json
{
  "phone": "+77001234567",
  "code": "1234",
  "new_password": "newSecret456"
}
```

Ответ 200: `{ "success": true, "message": "Пароль изменен" }`

Ошибки: 400 (код неверен/истёк, пароль слабый), 404 (аккаунт не найден).

### 3.7. `POST /mobile/auth/password/change` 🔒

Смена пароля авторизованным пользователем (Профиль → Сменить пароль).
Аккаунт определяется по Bearer-токену.

Запрос:

```json
{
  "current_password": "secret123",
  "new_password": "newSecret456"
}
```

Ответ 200: `{ "success": true, "message": "Пароль изменен" }`

Ошибки:

| HTTP | message |
|---|---|
| 400 | `Текущий пароль неверен` |
| 400 | `Пароль: минимум 6 символов, без пробелов` |
| 401 | — невалидный токен (клиент сделает refresh и повтор) |

### 3.8. `POST /mobile/auth/refresh`

Запрос:

```json
{ "refresh_token": "eyJhbGciOi..." }
```

Ответ 200:

```json
{
  "success": true,
  "access_token": "eyJhbGciOi...",
  "refresh_token": "eyJhbGciOi..."   // опционально: новая пара (ротация)
}
```

Если `refresh_token` не вернуть, клиент продолжит пользоваться старым.
Ошибка (refresh невалиден/отозван): 401 с любым телом → клиент разлогинит
пользователя с сообщением «Сессия истекла. Войдите заново».

### 3.9. `POST /mobile/auth/face-verify` 🔒

FaceID-сверка выполняется после чтения QR, непосредственно перед отметкой.
Живой снимок сравнивается с `Employee.photo` (S3), каждая попытка логируется.

Запрос:

```json
{
  "iin": "990101300123",
  "image": "<JPEG в base64, без префикса data:>",
  "qr_id": "park_karaganda_main_entrance"
}
```

Ответ 200 — успех:

```json
{
  "success": true,
  "match_percent": 94.5,
  "access_granted": true,
  "verification_token": "short-lived-one-time-token"
}
```

`verification_token` обязателен: одноразовый, TTL не более 2 минут, привязан к
сотруднику, `qr_id` и текущей сессии. После успешного `/attendance/scan` токен
сжигается; повторное использование запрещено.

Ответ 200 — лицо не совпало (это не HTTP-ошибка, а отрицательный результат):

```json
{
  "success": false,
  "access_granted": false,
  "match_percent": 41.2,
  "message": "Лицо не совпадает с профилем сотрудника"
}
```

Клиент принимает решение по `access_granted`; дает максимум **3 попытки**,
после чего возвращает пользователя к QR-сканеру без выхода из аккаунта.

Ошибки: 400 `Лицо на снимке не обнаружено`, 404 `У сотрудника нет фото в системе`.

## 4. Сотрудник

### 4.1. `GET /employees/{iin}/` 🔒

Профиль сотрудника. Эндпоинт уже существует в avatracker-back —
нужно разрешить доступ с мобильным JWT (или сделать зеркало `/mobile/employees/{iin}/`
с тем же ответом).

Ответ 200 (можно «как есть», можно обёрнутым в `{"data": {...}}` — клиент
поддерживает оба варианта):

```json
{
  "id": 1,
  "iin": "990101300123",
  "full_name": "Сериков Айдос Бекжанович",
  "photo": "https://s3.../employees/1.jpg",
  "active": true,
  "phone": "+77001234567",
  "position":  { "name": "Оператор аттракционов" },
  "division":  { "name": "Отдел аттракционов" },
  "employee_organization": { "name": "ТОО «AVATARIYA»" },
  "park_id": 5000011,
  "park_name": "AVATARIYA Karaganda",
  "city": "Караганда"
}
```

Допустимые вариации (клиент парсит толерантно): `photo_url` вместо `photo`;
`is_active` вместо `active`; `position`/`division`/`employee_organization`
строкой вместо объекта; `department` вместо `division`; `organization`
вместо `employee_organization`.

Если `position` или `division` возвращаются как GUID/UUID, нужно дополнительно
отдать человекочитаемые `position_name` и `division_name`. Клиент предпочитает
`*_name`, а UUID не показывает в профиле.

Поля `city` и `employee_organization`/`organization` можно возвращать, но
текущий экран «Профиль» их не показывает. В профиле видны: фото, ФИО, статус,
ИИН, телефон, должность, отдел и парк/филиал.

Поведение клиента, о котором важно знать:

- `active = false` → немедленный принудительный выход («Сотрудник неактивен»).
- `photo` пустое/нет → QR-отметка запрещена, пользователь увидит
  «В системе отсутствует фото сотрудника».
- 404 → выход с сообщением «Сотрудник не найден в системе».

## 5. Отметки и табель

### 5.1. `GET /mobile/qr-points/{qr_id}` 🔒

Справочная информация о точке отметки.
Текущая версия мобильного приложения **не вызывает** этот endpoint перед
отметкой: `qr_id` сразу передается в `face-verify`, затем в `attendance/scan`.
Endpoint можно оставить для будущего показа информации о QR-точке; он не входит
в минимальный набор для прод-теста.

Ответ 200:

```json
{
  "id": "park_karaganda_main_entrance",
  "name": "Главный вход",
  "park": "AVATARIYA Karaganda",
  "city": "Karaganda",
  "address": "пр. Республики 11, Караганда",
  "latitude": 49.8047,
  "longitude": 73.1094,
  "allowed_radius": 50,
  "is_active": true
}
```

Ошибки: 404 `QR-код не найден в системе`.

### 5.2. `POST /mobile/attendance/scan` 🔒

Отметка по QR. Сервер сам определяет тип отметки по счёту за день:
1-я — `check_in`, 2-я — `check_out`, последующие — `presence`.

QR может содержать либо чистый идентификатор (`park_karaganda_main_entrance`),
либо JSON вида `{"qr_id":"park_karaganda_main_entrance"}`. Клиент извлекает
строку `qr_id` и дальше отправляет только её.

Запрос (отправляет клиент):

```json
{
  "iin": "990101300123",
  "qr_id": "park_karaganda_main_entrance",
  "employee_latitude": 49.80471,
  "employee_longitude": 73.10952,
  "accuracy_meters": 12,
  "is_mock_location": false,
  "face_verification_token": "short-lived-one-time-token",
  "scanned_at": "2026-06-12T08:56:00+05:00"
}
```

`device_id` не отправляется и не требуется. Проверка, что отметку делает именно
этот сотрудник, выполняется через Bearer-сессию, `iin` и одноразовый
`face_verification_token`.

Серверные проверки: QR существует и активен; гаверсинус(координаты, точка) ≤
`allowed_radius` (50 м); `is_mock_location = false`; `accuracy_meters ≤ 50`;
сотрудник активен; FaceID-токен существует, не истек, относится к этому
сотруднику и QR и еще не использован. Антифрод-поля (`accuracy_meters`,
`is_mock_location`, результат FaceID) логировать.

Ответ 200 — успех:

```json
{
  "success": true,
  "mark_type": "check_in",
  "message": "Отметка успешно засчитана",
  "distance_meters": 12,
  "park": "AVATARIYA Karaganda",
  "scanned_at": "2026-06-12T08:56:03+05:00"
}
```

Бизнес-отказы — **HTTP 400 с телом результата** (клиент показывает их в шторке
результата, а не как сетевую ошибку):

```json
{
  "success": false,
  "error_code": "OUT_OF_RADIUS",
  "message": "Вы находитесь далеко от точки отметки",
  "distance_meters": 184,
  "allowed_radius": 50
}
```

| `error_code` | message | Когда |
|---|---|---|
| `OUT_OF_RADIUS` | `Вы находитесь далеко от точки отметки` | дистанция > радиуса (приложить `distance_meters`, `allowed_radius`) |
| `QR_INACTIVE` | `Эта точка отметки отключена` | `is_active = false` |
| `QR_NOT_FOUND` | `QR-код не найден в системе` | неизвестный `qr_id` |
| `MOCK_LOCATION` | `Обнаружена поддельная геолокация. Отметка запрещена` | `is_mock_location = true` |
| `LOW_ACCURACY` | `Низкая точность GPS. Выйдите на открытое место` | `accuracy_meters > 50` |
| `EMPLOYEE_INACTIVE` | `Доступ запрещен. Сотрудник неактивен` | сотрудник выключен |
| `FACE_REQUIRED` | `Подтвердите личность перед отметкой` | FaceID-токен отсутствует/истек/уже использован |

### 5.3. `GET /mobile/attendance/timesheet?iin={iin}&month={YYYY-MM}` 🔒

Табель за месяц: **все** дни месяца, включая будущие.

Ответ 200:

```json
{
  "month": "2026-06",
  "days": [
    {
      "date": "2026-06-01",
      "day_type": "working_day",
      "work_start": "09:00",
      "work_end": "18:00",
      "lunch_start": "13:00",
      "lunch_end": "14:00",
      "worked_minutes": 480,
      "remaining_minutes": 0,
      "status": "on_time",
      "place": "AVATARIYA Karaganda",
      "scans": [
        {
          "scanned_at": "2026-06-01T08:56:00+05:00",
          "mark_type": "check_in",
          "park": "AVATARIYA Karaganda"
        },
        {
          "scanned_at": "2026-06-01T18:02:00+05:00",
          "mark_type": "check_out",
          "park": "AVATARIYA Karaganda"
        }
      ]
    },
    {
      "date": "2026-06-06",
      "day_type": "weekend",
      "work_start": null,
      "work_end": null,
      "lunch_start": null,
      "lunch_end": null,
      "worked_minutes": 0,
      "remaining_minutes": 0,
      "status": "weekend",
      "place": "AVATARIYA Karaganda",
      "scans": []
    }
  ]
}
```

Правила полей:

- `day_type`: `working_day` | `weekend` (синоним `day_off` тоже понимается).
- `status`: `on_time` | `late` | `absent` | `weekend` | `weekend_work` | `no_scan`
  (`missed` понимается как `absent`).
- `worked_minutes` — за вычетом обеда; для дня «в процессе» (сегодня, ещё нет
  ухода) — фактически отработанное к текущему моменту, обед вычитается.
- `remaining_minutes` рассчитывается от индивидуальной длительности смены;
  для выходных — 0.
- `lunch_start`/`lunch_end` — только у рабочих дней (`null` в выходные); клиент
  показывает строку «Обед (не учитывается)».
- Будущие рабочие дни: `status = "no_scan"`, `worked_minutes = 0`, `scans = []`.

### 5.4. `GET /tardiness/?iin={iin}&period_from={YYYY-MM-DD}&period_to={YYYY-MM-DD}` 🔒

Аналитика опозданий за выбранный период. Экран «Аналитика» вызывает этот
endpoint для недели или месяца и больше не рассчитывает опоздания из табеля.

Параметры:

| Поле | Описание |
|---|---|
| `iin` | ИИН сотрудника |
| `period_from` | начало периода, формат `YYYY-MM-DD` |
| `period_to` | конец периода, формат `YYYY-MM-DD` |

Ответ 200:

```json
{
  "iin": "990101300123",
  "employee_name": "Тестовый Сотрудник",
  "schedule_name": "2/2",
  "schedule_start_time": "09:00:00",
  "period_from": "2026-06-01",
  "period_to": "2026-06-23",
  "count": 15,
  "max_tardiness": 178,
  "avg_tardiness": 62,
  "results": [
    {
      "date": "2026-06-23",
      "auth_time": "2026-06-23T09:07:50+05:00",
      "schedule_start_time": "09:00:00",
      "tardiness_minutes": 7
    }
  ]
}
```

Клиент дополнительно считает сумму минут опоздания из `results`.
Список `results` клиент сортирует по дате от новых к старым. `count`,
`max_tardiness` и `avg_tardiness` желательно возвращать всегда; если они
отсутствуют, клиент может восстановить часть значений из `results`, но это
только фолбэк.

## 7. Чек-лист реализации (важное поведение клиента)

1. **Login обязан вернуть `employee` или `iin`** — иначе вход на новом устройстве невозможен (§3.3).
2. `verify-code` сброса **не сжигает** код — его сожжёт следующий `reset` (§3.5).
3. Бизнес-отказы скана — HTTP 400 **с** `success/message/error_code` в теле, не «голый» 4xx (§5.2).
4. Все `message` — готовые русские тексты для пользователя (§1.2).
5. 401 без `message` клиент озвучит как «Сессия истекла» — для ошибок логина использовать 400 (§1.2).
6. `active=false` в любом ответе профиля → клиент сам разлогинит пользователя (§4.1).
7. Сотрудник без `photo` может войти, но не сможет сделать QR-отметку (§4.1, §3.9).
8. Табель возвращает индивидуальные `work_start/work_end`; фиксированной нормы в клиенте нет (§1.4, §5.3).
9. После `password/reset` отозвать refresh-токены аккаунта (§3.6).
10. Rate limit на SMS: 60 с между отправками, ≤5 проверок кода (§1.5).
11. `device_id` не нужен: валидация отметки идёт по Bearer-сессии, ИИН,
    QR, геолокации и снимку лица (§0.1, §1.7).
12. Экран «Аналитика» вызывает `/tardiness/`, а не `/mobile/attendance/statistics` и не считает опоздания из табеля (§5.4).
13. Face verification при входе не выполняется; снимок делается только перед
    QR-отметкой и уходит непосредственно в `/api/qr/scan/` (§0.1).

## 8. Демо-режим клиента (для сверки поведения)

APK собирается с `--dart-define=MOCK_API=true`: все ответы эмулируются на устройстве
(файл `lib/core/network/mock_interceptor.dart` — эталон ожидаемых ответов).
Демо-данные: аккаунт `+7 700 123 45 67` / `123456`, SMS-код всегда `1234`,
экран Face verification делает снимок камеры, QR со строкой `far` →
`OUT_OF_RADIUS`, со строкой `inactive` → `QR_INACTIVE`. Мок также отдаёт
табель и аналитику `/tardiness/`.

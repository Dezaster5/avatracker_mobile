# Техническое задание на разработку мобильного приложения **AvaTracker**

> **Архивная исходная версия.** Актуальные изменения MVP описаны в
> `API_CONTRACT.md` и `README.md`: FaceID выполняется перед каждой QR-отметкой,
> `device_id` не используется, раздел «Статистика» заменен клиентской
> «Аналитикой» опозданий и отработанного времени. При противоречиях приоритет
> имеет `API_CONTRACT.md`.
> Указанный ниже архивный `/api/v1/mobile/attendance/timesheet` отсутствует в
> production. Текущий табель строится из `/api/v1/employee-identification-list/`.
> Текущая версия исходного кода: `1.0.6+15`.

**Платформа:** Flutter
**Проект:** AvaTracker Mobile
**Цель:** учет рабочего времени сотрудников через QR-отметки, FaceID-проверку, SMS-авторизацию и контроль геолокации.

---

## 1. Общая логика приложения

Мобильное приложение **AvaTracker** нужно для сотрудников, чтобы они могли:

1. Зарегистрироваться по номеру телефона.
2. Подтвердить номер через SMS-код.
3. Пройти проверку по ИИН.
4. При входе проходить FaceID-сверку с базовой фотографией сотрудника.
5. Сканировать QR-код на рабочей локации.
6. Получать отметку прихода, ухода или присутствия.
7. Смотреть свой табель, историю отметок и статистику.

Админ-панель **AvaTracker.online** должна получать все отметки сотрудников и отображать их по подразделениям, паркам, городам, устройствам сканирования и QR-локациям.

## 3. Регистрация сотрудника

### 3.1. Экран регистрации

| Поле | Обязательное | Комментарий |
|---|---:|---|
| Номер телефона | Да | Формат `+7 XXX XXX XX XX` |
| ИИН | Да | 12 цифр |
| SMS-код | Да | После отправки кода |

---

### 3.2. Проверка сотрудника по ИИН

После ввода ИИН приложение делает GET-запрос:

```http
GET https://avatracker.online/api/v1/employees/{iin}
```

Пример:

```http
GET https://avatracker.online/api/v1/employees/642918307154
```

Система должна проверить:

- существует ли сотрудник;
- активен ли сотрудник;
- совпадает ли номер телефона, если он есть в базе;
- есть ли фотография сотрудника в AvaTracker;
- есть ли необходимые данные: ФИО, должность, отдел, филиал, подразделение.

---

### 3.3. Условия успешной регистрации

Регистрация разрешена, если:

- ИИН найден в базе;
- сотрудник активный;
- номер телефона валидный;
- SMS-код подтвержден;
- сотрудник не заблокирован;
- есть базовые данные сотрудника.

---

### 3.4. Условия отказа

| Ситуация | Текст ошибки |
|---|---|
| ИИН не найден | Сотрудник не найден в системе |
| Сотрудник неактивен | Доступ запрещен. Сотрудник неактивен |
| SMS-код неверный | Неверный SMS-код |
| Номер уже используется | Этот номер уже привязан к другому сотруднику |
| Нет доступа к серверу | Ошибка соединения. Попробуйте позже |

## 4. Подтверждение через SMS

### 4.1. Провайдер

Использовать SMS-провайдера:

```text
SMSc.kz
```

---

### 4.2. Логика

1. Пользователь вводит номер телефона.
2. Система генерирует одноразовый код.
3. Код отправляется через **SMSc.kz**.
4. Пользователь вводит код в приложении.
5. Backend проверяет код.
6. При успешной проверке создается сессия пользователя.

---

### 4.3. Требования к SMS-коду

| Параметр | Значение |
|---|---|
| Длина кода | 4–6 цифр |
| Срок действия | 3–5 минут |
| Повторная отправка | Через 60 секунд |
| Количество попыток | Не более 5 |
| Блокировка | После 5 неверных попыток |

## 5. Авторизация и FaceID

### 5.1. Логика входа

При каждом входе в мобильное приложение:

1. Пользователь открывает приложение.
2. Приложение определяет сохраненный ИИН.
3. Выполняется повторный GET-запрос:

```http
GET https://avatracker.online/api/v1/employees/{iin}
```

Пример:

```http
GET https://avatracker.online/api/v1/employees/642918307154
```

4. Проверяется, активен ли сотрудник.
5. Если сотрудник активен — запускается FaceID-сверка.
6. Если сотрудник неактивен — доступ запрещается.

---

### 5.2. FaceID-сверка

FaceID должен сверять лицо пользователя с фотографией сотрудника из базы AvaTracker.

Проверка идет по ИИН:

1. Получить данные сотрудника по ИИН.
2. Получить фото сотрудника.
3. Открыть камеру.
4. Сделать live-снимок.
5. Сравнить live-снимок с базовым фото.
6. При успешном совпадении открыть приложение.
7. При несовпадении запретить вход.

---

### 5.3. Защита от обхода

Нужно предусмотреть:

- запрет входа без FaceID;
- проверку live-камеры, чтобы нельзя было подставить фото с экрана;
- ограничение попыток;
- логирование неудачных попыток;
- повторную проверку активности сотрудника при каждом входе.

---

### 5.4. Ошибки FaceID

| Ситуация | Текст |
|---|---|
| Лицо не распознано | Не удалось распознать лицо. Попробуйте еще раз |
| Фото не совпадает | Лицо не совпадает с профилем сотрудника |
| Нет фото в базе | В системе отсутствует фото сотрудника |
| Сотрудник неактивен | Доступ запрещен. Сотрудник неактивен |
| Нет камеры | Нет доступа к камере |

## 7. QR-сканирование

### 7.1. Основная логика

QR-коды размещаются в разных парках, городах и локациях.

Каждый QR-код должен содержать уникальный идентификатор точки:

```json
{
  "qr_id": "park_karaganda_avatariya_main_entrance_001"
}
```

Через этот идентификатор backend должен понимать:

- город;
- парк;
- филиал;
- подразделение;
- точку сканирования;
- адрес;
- геолокацию QR-точки;
- допустимый радиус;
- активен ли QR-код.

---

### 7.2. Процесс отметки

1. Сотрудник нажимает кнопку сканирования.
2. Открывается камера.
3. Сотрудник сканирует QR-код.
4. Приложение запрашивает геолокацию устройства.
5. Backend получает:
   - ИИН сотрудника;
   - ID сотрудника;
   - QR ID;
   - координаты сотрудника;
   - время сканирования;
   - устройство;
   - IP;
   - тип действия.
6. Backend проверяет расстояние между сотрудником и QR-точкой.
7. Если сотрудник находится в радиусе до 50 метров — отметка засчитывается.
8. Если дальше 50 метров — отметка не засчитывается.

---

### 7.3. Геолокационная проверка

Обязательное условие:

```text
Расстояние между координатами QR-точки и координатами телефона сотрудника <= 50 метров
```

Если сотрудник не дал доступ к геолокации, сканирование запрещается.

---

### 7.4. Ошибки QR-сканирования

| Ситуация | Текст ошибки |
|---|---|
| Нет доступа к геолокации | Разрешите доступ к геолокации для отметки |
| Сотрудник далеко от QR | Вы находитесь далеко от точки отметки |
| QR не найден | QR-код не найден в системе |
| QR неактивен | Эта точка отметки отключена |
| Повторная отметка | Отметка уже была сделана |
| Нет интернета | Нет соединения с сервером |
| Сотрудник неактивен | Доступ запрещен. Сотрудник неактивен |

## 8. Типы отметок

Нужно определить бизнес-логику отметок.

Рекомендуемый вариант:

| Ситуация | Тип отметки |
|---|---|
| Первое сканирование за день | Приход |
| Второе сканирование за день | Уход |
| Дополнительные сканирования | Дополнительная отметка / Проверка присутствия |
| Сканирование в выходной | Работа в выходной |
| Сканирование вне графика | Внеплановая отметка |
| Опоздание | Опоздание |

Если у сотрудника график `09:00–18:00`, система должна считать:

- пришел вовремя;
- опоздал;
- ушел раньше;
- отработал полный день;
- не пришел;
- работал в выходной.

## 20. Логика отметки рабочего времени

### 20.1. Расчет рабочего дня

Система должна учитывать график сотрудника:

- рабочие дни;
- выходные;
- начало смены;
- конец смены;
- допустимое время опоздания, если есть;
- праздничные дни, если будут заведены.

---

### 20.2. Расчет опоздания

Если рабочее время начинается в `09:00`, а отметка в `09:01` и позже — статус `Опоздал`, если не задан grace period.

Можно добавить настройку:

```text
Допустимое опоздание: 5 минут
```

---

### 20.3. Расчет отработанного времени

Если есть приход и уход:

```text
Отработанное время = время ухода - время прихода
```

Если есть только приход:

```text
Отработанное время считается до текущего времени
```

Если отметок нет:

```text
Отработано: 0
Статус: Нет данных о сканированиях
```

## 25. Критерии приемки

Проект считается выполненным, если:

1. Сотрудник может зарегистрироваться по телефону и ИИН.
2. SMS-код приходит через SMSc.kz.
3. Неактивный сотрудник не может войти.
4. При каждом входе проверяется активность сотрудника.
5. FaceID сверяет лицо с фото из AvaTracker.
6. Без успешного FaceID вход запрещен.
7. QR-код успешно сканируется.
8. Отметка засчитывается только в радиусе 50 метров.
9. Если сотрудник дальше 50 метров — отметка отклоняется.
10. В табеле отображаются рабочие дни и история отметок.
11. В статистике отображается план / факт, опоздания и пропуски.
12. В профиле отображаются данные сотрудника.
13. В админке видны все мобильные отметки.
14. Админ может создавать и редактировать QR-точки.
15. Все ошибки логируются.
16. Приложение работает на iOS и Android.

## 26. Важные замечания для разработки

Критично не делать просто QR-сканер. Главная ценность системы — защита от невалидных отметок.

Минимальная связка проверки должна быть такой:

```text
ИИН сотрудника + активность в базе + FaceID + QR ID + геолокация телефона + расстояние до QR-точки
```

Если хотя бы один пункт не проходит — отметку не засчитывать.
